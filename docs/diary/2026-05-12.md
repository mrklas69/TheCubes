**(C) Pivot — user otázka:** *„Běží více instancí Claude nad tímto projektem. Jak se provádí, aby se předešlo nežádoucím konfliktům?"*

5 reálných rizik (od největšího):
1. **File-level race** v pracovním stromu — dvě instance editují stejný soubor, druhý zápis přepíše první (žádný OS lock, Edit tool chytí jen předchozí Read téže instance).
2. **`%BEGIN` stash race** — dvě paralelní `git add . && git stash` posloupnosti se rozpadnou (první stashne, druhá stashne nic, popování pak vede ke konfliktu nebo ztrátě).
3. **Docs race** — DIARY.md / TODO.md / MEMORY.md (index) — Write přepíše bez merge.
4. **Session numbering collision** — obě instance napíšou `## Sezení 24` v diáři.
5. **Port 8000** — bezpečné (jeden listener, ostatní vidí „už poslouchá").

Doporučení (od silného ke kompromisu):
- **`git worktree add ../TheCubes-<topic> feat/<topic>`** per instance — každá instance vlastní adresář + větev, konflikty se řeší až mergem. Nejčistší.
- **„Dirigent" pattern** — jedna instance smí docs + `%BEGIN`/`%END` + commits, ostatní jen kód v izolovaném souboru (např. `src/terrain.js` ve sdíleném worktree, kde nikdo jiný `src/` needitujе).
- **Minimum** — single instance pro `%BEGIN`/`%END` a Edit operace, paralelní jen `%THINK`/research.

**(D) Live demonstrace konfliktu.** Mezi `%BEGIN` a `%END` této instance paralelní Claude commitla na `feat/terrain` **dva commity**:
- `010d7ac refactor(wipe): odstranit factory toy + severskou diorámu (DD-32 Fáze 1)`

Mezi prvním a druhým `git status` v této instanci se strom přepnul z `M main.js + ?? terrain.js` → `clean`. **Přesně ten case, který jsme diskutovali, naživo.** Tahle instance dělala jen %BEGIN diagnostiku a meta-diskusi; veškerý kód sez. 24 patří paralelní instanci.
